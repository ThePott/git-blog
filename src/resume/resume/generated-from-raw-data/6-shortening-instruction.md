# 질문

질문 03.

끝이 좋았던 경험과 좋지 못했던 경험을 각각 소개해주세요. 특히 좋지 못했던 경험에서 무엇을 느꼈으며, 이후 유사한 상황에서
어떻게 다르게 대처하고자 했는지(혹은 대처했는지) 서술해주세요.

최대 1500자

# 대답

학원 강사 생활을 마무리한 날 밤, 평소 제 수업을 잘 따라오던 학생 두 명이 인사 메시지를 보내왔습니다. 제가 그저 그들의
인생에서 지나가는 한 사람이 아니라 메시지를 따로 적을 수고를 할 만한 사람이라는 것에 감사했습니다. 한 학생은 오랜 시간이
지나 다시 연락해 안부를 물었습니다. 학생은 저와의 수업이 종종 생각나고, 덕분에 더 잘하게 되었다 했습니다. 힘든 강사
생활을 저는 그렇게 보답받았습니다. 덕분에 그 이후에도 계속 누군가의 성장에 함께하고 싶다고 생각했습니다. "성장기회의
평등"을 위해 앞장서는 인프런과 함께 저도 그들의 성장에 기여하고 싶습니다.

오랜 기간 타인의 성장에 집중하며 지냈지만, 부트캠프 기간 동안엔 타인과 성장으로 경쟁했습니다. 부트캠프 심화반에서는 3달
동안 각자의 프로젝트를 개발했습니다. 마지막 수업에선 배포한 것으로 시연 발표를 하기로 하였습니다. 수업 직전에 배포를
하고 발표를 하였으나, 다른 수강생들은 서버로부터 응답을 못 받고 있었습니다. 발표 순서를 뒤로 미루고 디버깅을 하였으나
결국 이를 해결하지 못했습니다. 배포조차 제대로 못 했다는 것이 부끄러웠습니다.

문제는 크게 세 가지였습니다. 첫째, API URL을 일괄 교체하지 못했습니다. 데이터베이스를 MongoDB에서 Neon으로 교체한 뒤,
API URL을 교체했습니다만, 옛날 URL가 남아있는 곳도 있었습니다. 당시 오류의 원인을 이것으로 오판하여 진짜 원인을 발견하는
것이 늦어졌습니다.

둘째, 빌드 오류를 해결하는 데에 시간이 오래 걸렸습니다. 평소에 개발 서버에서만 작업하며 빌드를 하지 않았습니다. 그러다
배포 직전에 빌드를 했는데 수많은 오류가 발생했습니다. 개중에는 처음 보는 오류도 있어 바로 해결하는 게 어려웠습니다.

셋째, 배포 환경에서만 발생하는 오류를 해결해지 못했습니다. 배포를 마지막에 한 번만 하다보니 모든 오류가 낯설었고 발표의
압박 속에선 이를 해결할 수 없었습니다.

문제가 명확한 만큼 해결법도 명확했습니다. 이후 여러 곳에서 사용되는 것은 환경 변수로 관리했습니다. 덕분에 환경 변수 값만
바꾸며 여러 서버에서 테스트할 수 있었습니다.

또한 이후의 모든 프로젝트에서는 husky를 세팅하여 pre-push 훅으로 build 명령어를 추가하였습니다. 수시로 빌드를 하며
오류를 수정했습니다.

빌드처럼 배포도 주기적으로 하였습니다. 또한 배포 시에는 오류 발생을 상정하고 오류를 해결하는 시간으로 하루를 잡았습니다.
실제 Arbor The Tree를 작업할 때도 2주 간격의 미팅에 앞서, 언제나 미팅 전날 배포를 하고 당일엔 디버깅만 하였습니다.
배포에 충분한 시간을 들인 덕에 모든 미팅마다 진척 상황을 잘 전달할 수 있었습니다.

개발을 하며 "Fail Fast"라는 격언을 되새깁니다. 한 번에 완성하려다 실패한 위 경험 이후, 최대한 빨리 실패하고 이를
수정하는 루틴을 정착시켰습니다. 의도되지 않은 상황에서는 에러를 던지며 빨리 실패하고 확실하게 확인하며 오늘도 작업하고
있습니다.
