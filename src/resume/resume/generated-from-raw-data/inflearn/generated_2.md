### 지원 동기

입시 학원 강사로 4년 6개월 재직하며 학습 과정의 비효율을 직접 경험했습니다. 진도와 오답 관리를 수기로 동기화하는 데
학생당 5분, 수업당 50~100분이 소요되었고, 이 문제를 해결하고자 직접 웹앱(Arbor The Tree)을 제작했습니다.

인프런은 "성장기회의 평등"이라는 미션 아래 150만 명 이상의 사용자에게 학습 환경을 제공하고 있습니다. 저 역시 학습의
비효율을 기술로 해결한 경험이 있기에, 인프런의 미션에 깊이 공감합니다. 특히 학습자의 입장에서 문제를 정의하고, 기술로
해결책을 만들어본 경험이 인프런에서 가치를 만드는 데 기여할 수 있다고 생각합니다.

### 개선하고 싶은 부분: 강의 목차 네비게이션 UX

인프런에서 강의를 수강하며 느낀 불편함이 있습니다. 강의 목차가 길어질수록 현재 수강 중인 섹션을 찾기 어렵고, 특정 강의로
빠르게 이동하기 위해 스크롤을 반복해야 합니다.

**현재 문제점:**

- 강의 목차가 100개 이상인 경우, 현재 위치 파악이 어려움
- 섹션 간 이동 시 매번 스크롤이 필요
- 완료한 강의와 미완료 강의의 시각적 구분이 스크롤 없이는 한눈에 들어오지 않음

**개선 방안:**

1. **가상 스크롤 도입**: Arbor The Tree에서 TanStack Virtual을 적용해 그리드 입력 반응 시간을 89% 개선한 경험이
   있습니다. 강의 목차에도 가상 스크롤을 적용하면 DOM 노드 수를 줄여 렌더링 성능을 개선할 수 있습니다.

2. **현재 위치 고정 표시**: 스크롤과 무관하게 현재 수강 중인 강의 제목을 상단에 sticky로 표시하면, 사용자가 항상 자신의
   위치를 인지할 수 있습니다.

3. **섹션별 접기/펼치기 상태 기억**: 사용자가 접어둔 섹션 상태를 localStorage에 저장하면, 페이지 재방문 시에도 이전
   상태를 유지할 수 있습니다. Browser Storage 활용 경험을 바탕으로 구현할 수 있습니다.

4. **진행률 미니맵**: 전체 강의 진행 상황을 한눈에 보여주는 미니맵을 사이드바에 추가하면, 스크롤 없이도 완료/미완료
   강의를 파악할 수 있습니다. 이는 Arbor The Tree의 칸반 보드에서 진행 상황을 시각화한 경험과 유사합니다.

학습 도구를 직접 만들어본 경험을 바탕으로, 학습자의 관점에서 인프런 서비스의 사용자 경험을 개선하는 데 기여하고
싶습니다.

## 질문 2. 기술적 도전 과제와 Trade-off 분석

**개발 과정에서 마주했던 가장 기술적인 도전 과제를 소개해주세요. 단순히 새로운 기술을 도입한 경험보다는, 문제 해결을
위해 고려했던 여러 대안들의 장단점(Trade-off)을 어떻게 분석했고, 왜 최종적인 방식을 선택했는지 '논리적인 근거'를
중심으로 기술해주세요.**

---

### 도전 과제: PDF 생성 방식 선택

Arbor The Tree에서 학생의 오답을 모아 학습지(PDF)로 제작하는 기능이 필요했습니다. 80쪽 분량의 PDF를 생성해야 했고,
사용자 경험과 서버 부담을 모두 고려해야 했습니다.

### 고려한 대안들

**대안 1: react-pdf (클라이언트 사이드)**

- 장점: 서버 부담 없음, 실시간 미리보기 가능
- 단점: 브라우저 메인 스레드 점유로 UI 블로킹 발생, 80쪽 기준 생성 시간 긺

**대안 3: Typst (서버 사이드)**

- 장점: 컴파일 기반으로 빠른 생성 속도, 메모리 효율적, 수학 수식 렌더링에 최적화
- 단점: 학습 곡선 존재, npm 패키지로 설치 불가

### Trade-off 분석

| 기준           | react-pdf         | Puppeteer | Typst         |
| -------------- | ----------------- | --------- | ------------- |
| 생성 속도      | 느림              | 보통      | 빠름          |
| 메모리 사용    | 낮음 (클라이언트) | 높음      | 낮음          |
| UI 블로킹      | 있음              | 없음      | 없음          |
| 배포 복잡도    | 낮음              | 보통      | 높음          |
| 수학 수식 지원 | 제한적            | CSS 의존  | 네이티브 지원 |

### 최종 선택: Typst

**선택 근거:**

1. **사용자 경험 우선**: 브라우저 메인 스레드 점유 없이 PDF를 생성해야 했습니다. react-pdf는 80쪽 생성 중 UI가 멈추는
   문제가 있었습니다.

2. **배포 환경 제약**: Railway 무료 플랜의 메모리 제한(512MB)으로 Puppeteer는 부적합했습니다.

3. **도메인 적합성**: 수학 학원용 학습지이므로 수식 렌더링이 중요했고, Typst는 LaTeX 수준의 수식을 네이티브로
   지원합니다.

4. **성능 측정 결과**: 80쪽 기준 Typst가 react-pdf 대비 30.7% 빠른 생성 시간을 보였습니다.

**해결한 문제:**

- npm install로 설치되지 않는 Typst compiler와 font 파일을 Railway 배포 시 자동 설치하는 커스텀 배포 스크립트를
  작성했습니다.
- 이를 통해 배포 복잡도 문제를 해결하고, Typst의 장점만 취할 수 있었습니다.

### 배운 점

기술 선택은 단순히 "최신 기술"이나 "익숙한 기술"이 아니라, 문제의 제약 조건(메모리, UX, 도메인)을 명확히 정의하고 각
대안의 Trade-off를 분석한 후 결정해야 한다는 것을 배웠습니다.

## 질문 3. 끝이 좋았던 경험과 좋지 못했던 경험

**끝이 좋았던 경험과 좋지 못했던 경험을 각각 소개해주세요. 특히 좋지 못했던 경험에서 무엇을 느꼈으며, 이후 유사한
상황에서 어떻게 다르게 대처하고자 했는지(혹은 대처했는지) 서술해주세요.**

---

### 끝이 좋았던 경험: 서버 응답 속도 22배 개선

Arbor The Tree 개발 중 문제집 생성 API의 응답 시간이 7초에 달하는 문제가 있었습니다.

**문제 분석:**

- Neon DB(무료 플랜)의 cold start로 인한 지연
- API 서버(Railway, Singapore)와 DB(Neon, US East)의 지역 불일치로 인한 network latency

**해결 과정:**

1. Railway region을 US East에서 Singapore로 변경 → 여전히 5~7초
2. Neon의 무료 플랜이 cold start를 강제한다는 사실 확인
3. Railway에 PostgreSQL을 직접 배포, API와 동일 internal network에 위치

**결과:**

- 응답 시간 7초 → 0.3초 (22배 향상)
- 사용자가 체감할 수 있는 수준의 성능 개선 달성

**좋았던 이유:** 문제의 원인을 단계적으로 분석하고, 각 단계에서 측정을 통해 가설을 검증했습니다. 단순히 "느리다"가
아니라 "왜 느린지"를 파악하고 해결한 경험이었습니다.

---

### 끝이 좋지 못했던 경험: 프로젝트 중단

Arbor The Tree는 전 재직 학원 원장과의 계약으로 시작한 프로젝트였습니다. 부트캠프 종료 직후 원장에게 먼저 제안하여
프로젝트를 시작했고, 4개월간 개발을 진행했습니다.

**문제 상황:**

- 개발 완료 직전, 원장의 일방적인 계약 취소 통보
- 모바일 반응형 구현(진도표, 오답체크 페이지)을 앞두고 있었음
- 실제 사용자(학생, 강사) 피드백을 받기 직전 단계에서 중단

**느낀 점:**

1. **계약의 중요성**: 구두 약속에 의존했던 점이 문제였습니다. 개발 범위, 일정, 비용, 중단 조건 등을 문서화하지
   않았습니다.

2. **MVP의 중요성**: 4개월간 "완벽한 제품"을 만들려 했지만, 더 빨리 MVP를 배포하고 사용자 피드백을 받았어야 했습니다.
   계약 취소 전에 가치를 증명할 기회가 있었을 수도 있습니다.

3. **의존성 분산**: 단일 클라이언트(원장)에게 전적으로 의존한 구조였습니다.

**이후 대처:**

1. **프로젝트 공개**: 중단되었지만 GitHub에 소스 코드를 공개하고, 포트폴리오로 활용했습니다. 기술적 성과(22배 응답 속도
   개선, 89% 렌더링 개선)는 여전히 유효합니다.

2. **문서화 습관**: 이후 어떤 협업에서든 범위, 일정, 조건을 먼저 문서화하는 습관을 들였습니다.

3. **작게 시작하기**: 다음 프로젝트에서는 MVP를 빠르게 배포하고, 사용자 피드백을 받으며 점진적으로 개선하는 방식을
   택하겠습니다.

**배운 점:** 프로젝트의 성공은 기술적 완성도만으로 결정되지 않습니다. 이해관계자와의 명확한 합의, 빠른 가치 증명, 리스크
분산이 함께 필요합니다. 이 경험은 아쉽지만, 다음 프로젝트에서 같은 실수를 반복하지 않기 위한 교훈이 되었습니다.
