# 인프런 프론트엔드 개발자 지원 - 추가 질문 답변

## 질문 1. 지원 동기 및 개선하고 싶은 부분

입시 학원 강사로 4년 6개월 재직하며 학습 과정의 비효율을 직접 경험했습니다. 진도와 오답 관리를 수기로 동기화하는 데
학생당 5분, 수업당 50~100분이 소요되었고, 이 문제를 해결하고자 직접 웹앱(Arbor The Tree)을 제작했습니다.

인프런은 "성장기회의 평등"이라는 미션 아래 150만 명 이상의 사용자에게 학습 환경을 제공하고 있습니다. 저 역시 학습의
비효율을 기술로 해결한 경험이 있기에, 인프런의 미션에 깊이 공감합니다. 특히 학습자의 입장에서 문제를 정의하고, 기술로
해결책을 만들어본 경험이 인프런에서 가치를 만드는 데 기여할 수 있다고 생각합니다.

인프런과 함께 하며 작은 것에서부터 기여를 하고 싶습니다. 인프런을 이용하며 여기는 이렇게 하면 되지 않을까 싶은 부분들이
있었습니다. 제가 생각할 수 있었던 만큼 인프런의 선배분들께서 이미 생각해보셨을 텐데 어떤 생각으로 혹은 과정으로 현재의
상태를 유지하고 있는 것인지 궁금합니다. 많이 듣고 많이 배우고 싶습니다. 그 와중에 제 생각이 맞는 부분이 있다면, 제가
떠올렸던 해결책으로 더 나은 사용자 경험을 제공할 수 있도록 인프런을 개선하고 싶습니다.

- layout shift 해결
    - `scrollbar-gutter:stable`로 스크롤바 유무와 무관하게 컨텐츠 위치 유지
        - 현재의 랜딩 페이지에서는 스켈레톤은 스클롤바가 없고, 로딩이 완료되면 스크롤바가 생김.
        - 스크롤바의 존재 여부에 따라 자식이 차지할 수 있는 너비가 달라짐 (MacOs "스크롤 막대 보기: 항상" 기준)
        - `scrollbar-gutter:stable`을 적용하여 스크롤바의 자리를 항상 유지하여 전체 페이지의 너비를 일정히 유지 가능
    - `<main/>`의 최소 높이를 설정하여 강의 로딩이 안 되었을 때도 푸터가 최초 뷰포트 밑에 있게 함
        - 현재 랜딩 페이지에서 로딩이 덜 되었을 때는 푸터가 보였다가 로딩이 끝나며 밑으로 내려가 안 보임.
        - 푸터의 짙은 색깔 때문에 깜빡임이 두드러짐
        - 로딩 여부와 무관하게 스크롤을 해야만 푸터가 보이게 하면 깜빡임을 완화할 수 있음

- 필터 기능 일원화
    - 현재는 최상단에서 검색, 중단에서 카테고리 선택으로 필터링할 수 있음. 본질적으로 필터를 한다는 점에서 같은데
      위치적으로 분리되어있음
    - 위치 뿐 아니라 기능적으로도 격리되어 있음. 제한된 카테고리에서 검색하는 것이 불가능 (예: "데이터 사이언스"
      카테고리에서 "초급" 검색 불가능)
    - 최상단에 검색창과 카테고리 선택을 같이 두고 두 필터를 연동하면 사용자 경험이 개선될 것으로 예상됨

- 사이드바를 사용하여 데스크탑에 최적화된 UI로 변경
    - 긴 글의 목차를 사이드바에 보여줘 글의 흐름을 더 쉽게 파악할 수 있게
        1. `querySelectorAll("h1, h2, ...")` 로 모든 헤딩 추출
        2. 사이드바에서의 제목을 클릭하면 해당하는 제목이 상단에 보이게 자동 스크롤
            - 이를 위해선 마크다운을 html로 변환할 때 heading에 id를 부여해서 돔 노드를 특정할 수 있게 해야 함

- 무한 스크롤로 사용자 체류 시간 증대
    - 무한 스크롤로 사용자 경험이 끊어지지 않게
    - 가상 스크롤로 돔 노드를 관리하여 성능 최적화
    - 푸터를 최소화하여 항상 노출
        - 무한 스크롤 적용시, 푸터를 보여줄 수 없는 문제 발생
        - 법적 의무사항으로 보기 쉬운 곳에 배치해야 하는 "이용 약관", "개인 정보 처리 방침" 등 최소한만 항상 노출
        - 펼치기 버튼 누르면 전체 푸터가 보이게 함

- 뷰포트에 따른 조건부 문의하기 위젯
    - 현재 별도 페이지로 넘어감
    - 데스크탑에서는 플로팅 위젯으로 뜨는 게 더 자연스러울 것
    - 다만 모바일이라면 화면이 작기에 현재의 리디렉션이 좋음
    - 클릭시 뷰포트 확인 -> 좁으면 리다이렉트, 넓으면 플로팅 위젯

인프런의 사용자의 입장에서 문제를 찾고, 인프런의 사용자가 만족할만한 해결책을 만들겠습니다.

## 질문 2. 기술적 도전 과제와 Trade-off 분석

**개발 과정에서 마주했던 가장 기술적인 도전 과제를 소개해주세요. 단순히 새로운 기술을 도입한 경험보다는, 문제 해결을
위해 고려했던 여러 대안들의 장단점(Trade-off)을 어떻게 분석했고, 왜 최종적인 방식을 선택했는지 '논리적인 근거'를
중심으로 기술해주세요.**

PDF는 브라우저와 무관한데 왜 클라이언트에서 생성해야 하지?

Arbor The Tree에서 학생의 오답을 모아 학습지(PDF)로 제작하는 기능이 필요했습니다. 한 학생의 오답 과제는 10쪽 미만인
경우가 보통이지만, MVP 개발 이후엔 한 반의 모든 오답과제를 한 번에 생성하는 기능을 제작할 계획이었습니다. 이를 위해선
100쪽 이상의 PDF를 제작할 수도 있어야 했습니다. 고객(원장)은 빠른 반응 속도를 요구하였기에, PDF 생성 과정에서 사용자
경험이 끊기지 않는 것이 가장 중요하였습니다.

React-PDF도 마음에 안 들고 Puppeteer도 마음에 안 든다

자료를 조사하며 두 가지 후보가 눈에 들어왔습니다. 클라이언트 사이드에서 html 문서를 PDF로 변환하는 React-PDF, 그리고
서버 사이드에서 html을 PDF로 변환할 수 있게 해주는 Puppeteer였습니다. 여기서 Puppeteer는 빠르게 후보에서 제외되었습니다.
PDF 생성은 해당 라이브러리의 곁가지 기능일 뿐이다보니, PDF 생성만을 위해서 이것을 쓰는 것은 닭 잡는 데에 소 잡는 칼 쓰는
격이었습니다.

React-PDF에도 명확한 문제점이 보였습니다. 첫째로 리액트 컴포넌트를 pdf로 변환한다는 것이었습니다. 오답 과제에는 한
페이지에 문제집 제목, 틀린 문제의 쪽 번호와 문제 번호만 들어가면 됩니다. 웹앱에서 보이는 화면을 PDF로 만드는 게 아니라
그저 문서를 만들 뿐이기에 돔 노드와는 무관했습니다. 둘째로 PDF 생성을 클라이언트 사이드에서 진행합니다. 브라우저는
기본적으로 싱글 스레드이기 때문에 오래 걸리는 작업을 하게 되면 화면이 멈추게 됩니다. 이를 해결하려면 웹 워커를 써서
복수의 스레드를 사용해야 합니다. 하지만 PDF 생성은 브라우저와 무관한데 이걸 웹 워커를 사용하면서까지 클라이언트
사이드에서 할 필요는 없다고 판단했습니다.

이에 리액트 바깥으로 시야를 옮겼습니다. PDF 생성 자체만을 검색하였고 이번에도 두 후보를 찾았습니다. 논문을 전문적으로
작성할 때 쓰인 LaTex, 그리고 더 나은 사용자경험을 제공하지만 대중적이진 않아 협업엔 부적합한 Typst였습니다. 둘 다 수학
수식, 그래프 등을 작성하는데 특화되었으나, Typst가 컴파일 속도가 더 빠르고 문법도 간단했습니다. 속도가 최우선이었기에
Typst를 택했습니다.

이상적으론 두 방법으로 PDF를 생성하고 생성 시간을 비교해야 했으나, 촉박한 개발 기간 하에서 낯선 기술 둘 모두를 익힐 순
없었습니다. React-PDF의 공식 문서에서 30 페이지 이상에선 웹 워커 사용을 권장하고 있다는 점, 그리고 서버 사이드에서
Typst로 PDF를 컴파일하면 생성 시간 동안 브라우저 메인 스레드를 전혀 점유하지 않는다는 점에 집중했습니다. 이로써 Typst
방법을 채택하기로 결정했습니다.

Railway에 배포 시 Typst 컴파일러와 폰트가 자동으로 설치되지가 않았습니다. 아래와 같은 문제 때문이었습니다.

- Typst 컴파일러는 npm을 지원하지 않음
- Typst 한글 폰트는 1. 로컬에 폰트 파일이 있어야 하고 2. typst root file보다 바깥에(src/asset) 있으면 안 됨

이에 Typst compiler와 font 파일을 Railway 배포 시 자동 설치하는 커스텀 배포 스크립트를 작성했습니다. 이를 통해 배포
복잡도 문제를 해결하고, Typst의 장점만 취할 수 있었습니다.

MVP를 완성한 뒤에 시간적 여유가 생겨 React-PDF로도 기능을 구현하여 두 방법의 속도를 비교하였습니다. 결과는 최초의
분석대로였습니다.

- 8쪽 생성 시간 4.7% 단축
- 80쪽 생성 시간 30.7% 단축
- 800쪽 생성 시간 61.3% 단축

"right tool for the right job"은 "최신 기술" 혹은 "익숙한 기술"이 되기 십상입니다. 그러나 가장 중요한 건 문제의 본질을
파악하는 것입니다.

### 도전 과제: PDF 생성 방식 선택

Arbor The Tree에서 학생의 오답을 모아 학습지(PDF)로 제작하는 기능이 필요했습니다. 80쪽 분량의 PDF를 생성해야 했고,
사용자 경험과 서버 부담을 모두 고려해야 했습니다.

### 고려한 대안들

**대안 1: react-pdf (클라이언트 사이드)**

- 장점: 서버 부담 없음, 실시간 미리보기 가능
- 단점: 브라우저 메인 스레드 점유로 UI 블로킹 발생, 80쪽 기준 생성 시간 긺

**대안 3: Typst (서버 사이드)**

- 장점: 컴파일 기반으로 빠른 생성 속도, 메모리 효율적, 수학 수식 렌더링에 최적화
- 단점: 학습 곡선 존재, npm 패키지로 설치 불가

### Trade-off 분석

| 기준           | react-pdf         | Puppeteer | Typst         |
| -------------- | ----------------- | --------- | ------------- |
| 생성 속도      | 느림              | 보통      | 빠름          |
| 메모리 사용    | 낮음 (클라이언트) | 높음      | 낮음          |
| UI 블로킹      | 있음              | 없음      | 없음          |
| 배포 복잡도    | 낮음              | 보통      | 높음          |
| 수학 수식 지원 | 제한적            | CSS 의존  | 네이티브 지원 |

### 최종 선택: Typst

**선택 근거:**

1. **사용자 경험 우선**: 브라우저 메인 스레드 점유 없이 PDF를 생성해야 했습니다. react-pdf는 80쪽 생성 중 UI가 멈추는
   문제가 있었습니다.

2. **배포 환경 제약**: Railway 무료 플랜의 메모리 제한(512MB)으로 Puppeteer는 부적합했습니다.

3. **도메인 적합성**: 수학 학원용 학습지이므로 수식 렌더링이 중요했고, Typst는 LaTeX 수준의 수식을 네이티브로
   지원합니다.

4. **성능 측정 결과**: 80쪽 기준 Typst가 react-pdf 대비 30.7% 빠른 생성 시간을 보였습니다.

**해결한 문제:**

- npm install로 설치되지 않는 Typst compiler와 font 파일을 Railway 배포 시 자동 설치하는 커스텀 배포 스크립트를
  작성했습니다.
- 이를 통해 배포 복잡도 문제를 해결하고, Typst의 장점만 취할 수 있었습니다.

### 배운 점

기술 선택은 단순히 "최신 기술"이나 "익숙한 기술"이 아니라, 문제의 제약 조건(메모리, UX, 도메인)을 명확히 정의하고 각
대안의 Trade-off를 분석한 후 결정해야 한다는 것을 배웠습니다.

## 질문 3. 끝이 좋았던 경험과 좋지 못했던 경험

**끝이 좋았던 경험과 좋지 못했던 경험을 각각 소개해주세요. 특히 좋지 못했던 경험에서 무엇을 느꼈으며, 이후 유사한
상황에서 어떻게 다르게 대처하고자 했는지(혹은 대처했는지) 서술해주세요.**

---

### 끝이 좋았던 경험: 서버 응답 속도 22배 개선

Arbor The Tree 개발 중 문제집 생성 API의 응답 시간이 7초에 달하는 문제가 있었습니다.

**문제 분석:**

- Neon DB(무료 플랜)의 cold start로 인한 지연
- API 서버(Railway, Singapore)와 DB(Neon, US East)의 지역 불일치로 인한 network latency

**해결 과정:**

1. Railway region을 US East에서 Singapore로 변경 → 여전히 5~7초
2. Neon의 무료 플랜이 cold start를 강제한다는 사실 확인
3. Railway에 PostgreSQL을 직접 배포, API와 동일 internal network에 위치

**결과:**

- 응답 시간 7초 → 0.3초 (22배 향상)
- 사용자가 체감할 수 있는 수준의 성능 개선 달성

**좋았던 이유:** 문제의 원인을 단계적으로 분석하고, 각 단계에서 측정을 통해 가설을 검증했습니다. 단순히 "느리다"가
아니라 "왜 느린지"를 파악하고 해결한 경험이었습니다.

---

### 끝이 좋지 못했던 경험: 프로젝트 중단

Arbor The Tree는 전 재직 학원 원장과의 계약으로 시작한 프로젝트였습니다. 부트캠프 종료 직후 원장에게 먼저 제안하여
프로젝트를 시작했고, 4개월간 개발을 진행했습니다.

**문제 상황:**

- 개발 완료 직전, 원장의 일방적인 계약 취소 통보
- 모바일 반응형 구현(진도표, 오답체크 페이지)을 앞두고 있었음
- 실제 사용자(학생, 강사) 피드백을 받기 직전 단계에서 중단

**느낀 점:**

1. **계약의 중요성**: 구두 약속에 의존했던 점이 문제였습니다. 개발 범위, 일정, 비용, 중단 조건 등을 문서화하지
   않았습니다.

2. **MVP의 중요성**: 4개월간 "완벽한 제품"을 만들려 했지만, 더 빨리 MVP를 배포하고 사용자 피드백을 받았어야 했습니다.
   계약 취소 전에 가치를 증명할 기회가 있었을 수도 있습니다.

3. **의존성 분산**: 단일 클라이언트(원장)에게 전적으로 의존한 구조였습니다.

**이후 대처:**

1. **프로젝트 공개**: 중단되었지만 GitHub에 소스 코드를 공개하고, 포트폴리오로 활용했습니다. 기술적 성과(22배 응답 속도
   개선, 89% 렌더링 개선)는 여전히 유효합니다.

2. **문서화 습관**: 이후 어떤 협업에서든 범위, 일정, 조건을 먼저 문서화하는 습관을 들였습니다.

3. **작게 시작하기**: 다음 프로젝트에서는 MVP를 빠르게 배포하고, 사용자 피드백을 받으며 점진적으로 개선하는 방식을
   택하겠습니다.

**배운 점:** 프로젝트의 성공은 기술적 완성도만으로 결정되지 않습니다. 이해관계자와의 명확한 합의, 빠른 가치 증명, 리스크
분산이 함께 필요합니다. 이 경험은 아쉽지만, 다음 프로젝트에서 같은 실수를 반복하지 않기 위한 교훈이 되었습니다.
