Arbor The Tree에서 학생의 오답을 모아 학습지(PDF)로 제작하는 기능이 필요했습니다. 100쪽 이상의 PDF도 생성할 수 있어야
했고, 고객은 빠른 반응 속도를 요구했기에 PDF 생성 중에도 UX가 끊기지 않는 것이 핵심이었습니다.

세 가지 후보를 검토했습니다.첫째, Puppeteer(서버 사이드)입니다. 브라우저 자동화 도구로 PDF 생성은 곁가지 기능이었습니다.
PDF 생성만을 위해 도입하기엔 과도한 선택이라 제외했습니다. 둘째, React-PDF(클라이언트 사이드)입니다. 두 가지 문제가
있었습니다. 오답 과제는 문제집 제목과 문제 번호만 필요해 리액트 컴포넌트 변환이 불필요했습니다. 또한 싱글 스레드인
브라우저에서 PDF를 생성하면 UI가 멈춥니다. 공식 문서에서도 30 페이지 이상에선 Web Worker 사용을 권장하고 있었습니다.
셋째, Typst(서버 사이드)입니다. 문서 조판 전용 도구로 LaTex보다 컴파일 속도가 빠르고 문법이 간결했습니다. 서버에서
컴파일하면 브라우저 메인 스레드를 전혀 점유하지 않아 UX 문제를 근본적으로 해결할 수 있었습니다.

Typst를 선택했지만 배포 이슈가 있었습니다. Railway 환경에서 Typst 컴파일러(npm 미지원)와 한글 폰트가 자동 설치되지
않았습니다. 이를 해결하기 위해 커스텀 배포 스크립트를 작성하여 자동화했습니다.

MVP 완성 후 React-PDF로도 구현하여 속도를 비교했습니다. Typst를 사용하면 생성 속도가 8쪽 4.7%, 80쪽 30.7%, 800쪽 61.3%
짧았고, 생성 중 UI 블로킹이 전혀 없었습니다.

"right tool for the right job"은 자칫 "최신 기술" 혹은 "익숙한 기술"이 되기 쉽습니다. "브라우저와 무관한 작업을 왜
클라이언트에서 해야 하는가"라는 본질을 파악하는 것이 가장 중요했습니다.
