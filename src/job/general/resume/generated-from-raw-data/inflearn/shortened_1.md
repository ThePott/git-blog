pdf는 브라우저와 무관한데 왜 클라이언트에서 생성해야 하지?

Arbor The Tree에서 학생의 오답을 모아 학습지(PDF)로 제작하는 기능이 필요했습니다. MVP 이후엔 한 반의 모든 오답과제를 한
번에 생성하는 기능을 제작할 계획이었고, 100쪽 이상의 PDF도 생성할 수 있어야 했습니다. 고객(원장)은 빠른 반응 속도를
요구했기에, PDF 생성 과정에서 사용자 경험이 끊기지 않는 것이 가장 중요했습니다.

React-PDF도 마음에 안 들고 Puppeteer도 마음에 안 든다

두 가지 후보가 있었습니다. 클라이언트에서 html을 pdf로 변환하는 React-PDF와 서버에서 변환하는 Puppeteer였습니다.
Puppeteer는 PDF 생성이 곁가지 기능이라 과도한 선택이었기에 제외했습니다.

React-PDF에도 문제가 있었습니다. 첫째, 리액트 컴포넌트를 pdf로 변환하는데, 오답 과제는 문제집 제목과 문제 번호만 필요해
돔 노드와 무관했습니다. 둘째, 클라이언트에서 PDF를 생성하면 싱글 스레드인 브라우저가 멈춥니다. 웹 워커로 해결할 수
있지만, 브라우저와 무관한 PDF 생성을 굳이 클라이언트에서 할 필요가 없다고 판단했습니다.

리액트 바깥으로 시야를 옮겨 LaTex와 Typst를 찾았습니다. 둘 다 수학 수식, 그래프 작성에 특화되었으나, Typst가 컴파일
속도가 빠르고 문법도 간단해 Typst를 택했습니다.

이상적으론 두 방법을 비교해야 했으나, 촉박한 기간에 낯선 기술 둘 모두를 익힐 순 없었습니다. React-PDF는 30페이지
이상에서 웹 워커 사용을 권장하고, 서버에서 Typst로 컴파일하면 브라우저 메인 스레드를 전혀 점유하지 않는다는 점에 집중해
Typst를 채택했습니다.

Railway 배포 시 Typst 컴파일러와 폰트가 자동 설치되지 않았습니다.

- Typst 컴파일러는 npm을 지원하지 않음
- Typst 한글 폰트는 로컬에 있어야 하고, typst root file보다 바깥에 있으면 안 됨

이에 Railway 배포 시 자동 설치하는 커스텀 배포 스크립트를 작성해 배포 복잡도 문제를 해결했습니다.

MVP 완성 후 React-PDF로도 구현하여 속도를 비교했습니다.

- 8쪽 생성 시간 4.7% 단축
- 80쪽 생성 시간 30.7% 단축
- 800쪽 생성 시간 61.3% 단축

"right tool for the right job"은 "최신 기술" 혹은 "익숙한 기술"이 되기 십상입니다. 그러나 가장 중요한 건 문제의 본질을
파악하는 것입니다.
