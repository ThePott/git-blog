# 페이타랩에 지원하는 이유는 무엇인가요?

페이타랩의 일하는 방식과 제 개발 철학이 같은 결이라 지원합니다.

## "왜?"를 묻는 본질 추구

- 가치관: 트렌드보다 본질, 복잡함보다 명확함
- Arbor The Tree 외주 프로젝트에서 "본질 정의 → 단순 해결책 → 측정" 사이클로 다음 성과 달성
    - PDF 생성: "PDF는 브라우저와 무관한 문서"라는 본질 정의 → 서버 Typst 컴파일로 전환 → react-pdf 대비 800쪽 기준 생성
      시간 61.3% 단축
    - 서버 응답: 7초 지연의 원인을 cold start + far region(Neon N.Virginia)으로 분리 → Railway 단일 리전에 PostgreSQL
      직접 배포 → 315ms로 22배 향상
    - 스프레드시트 UI: 필요한 건 스프레드시트가 아니라 자동완성·유효성 검사가 가능한 테이블 → TanStack Table로 직접
      구현, 무거운 의존성 제거
- 페이타랩의 "그냥 하는 일은 없다. 항상 '왜?'를 생각한다"와 일치

## 자영업자 문제를 직접 풀어본 경험

- 학원 원장(자영업자)의 운영 비효율을 외주 프로젝트로 해결
    - 강사 기억에 의존하던 진도 추적 → 칸반 보드로 시각화
    - 수업당 50~100분 걸리던 수동 동기화 → 오답 체크 자동 반영으로 0에 수렴
    - 인수인계가 어려운 CLI 시스템 → 아르바이트생도 즉시 쓸 수 있는 웹앱
- 패스오더가 풀고자 하는 사장님의 인건비·매출 문제와 같은 결의 문제를 직접 풀어본 경험이 도메인 적응에 도움이 되리라
  생각합니다.

## "빠르게 실행, 개선" 문화 적합

- 부트캠프 마지막 날 배포 실패로 시연하지 못한 경험을 기점으로, 외주 프로젝트에서는 2주 간격 미팅마다 기능 추가 없이
  배포만 진행하는 루틴을 정착시켰습니다.
- 깡통 배포 우선, 점진 개선이 기본 자세입니다.
- 페이타랩의 "완벽한 결과물은 없다, 빠르게 실행해 본다"와 일치합니다.

# 페이타랩에 입사 시 어떤 방식 및 계획으로 업무를 진행하고 싶으신가요?

"본질 정의 → 단순 해결책 → 측정 → 빠른 배포" 사이클을 패스오더에 적용하겠습니다.

## 1. 첫 1~3개월: 도메인·코드베이스 빠른 흡수

- 패스오더 앱과 백오피스를 직접 사용하며 사용자·점주 흐름 체화
- 기존 React/Angular 코드 컨벤션·아키텍처 파악, 작은 백오피스 버그 수정 PR로 첫 기여
- 미경험 영역(Angular, Monorepo, Jest)은 입사 전부터 공식 문서·실습으로 격차 좁히기 시작

## 2. 업무 흐름: "왜?"부터 시작

요구사항을 받으면 다음 순서로 진행합니다.

- 본질 정의: 어떤 사용자가, 어떤 상황에서, 어떤 비용을 치르는가
- 단순 해결책 설계: 라이브러리 도입 전 자체 구현 가능성 검토
- 측정 지표 설정: 응답 시간, 렌더링 시간, 번들 크기
- 작게 배포 후 검증 (A/B 테스트 포함)

Arbor에서 서버 응답 22배 향상(7초 → 315ms), Typst vs react-pdf A/B 테스트로 800쪽 기준 61.3% 단축을 검증한 경험을
패스오더 성능 개선에 그대로 적용하겠습니다.

## 3. 빠른 배포·코드 리뷰·지식 공유

- 큰 기능도 깡통 배포 후 점진 개선 (2주 배포 루틴 유지)
- 코드 리뷰에 적극 참여, 동료 리뷰 기준을 본인 코드 점검 체크리스트로 내재화
- 학습 내용을 사내 노션·팀 블로그로 공유

## 4. 우대사항 격차 보완 계획

- 테스트 코드: 백오피스 버그 수정 PR부터 Jest + React Testing Library 적용, 회귀 방지 목적의 최소 단위 테스트로 시작
- 성능 프로파일링: Chrome DevTools Performance/Network, React Profiler로 메모리 누수·불필요한 리렌더 진단을 루틴화
- 웹 접근성·표준: semantic 태그, img alt, link/button 구분, 키보드 네비게이션을 PR 기본 체크리스트로 운영
